🐧 My Daily Driver: Fedora KDE Plasma as a Computer Engineering Graduate

10 min read Linux
Table of Contents
Share: Twitter LinkedIn Copy Link

As a Computer Engineering graduate, I don't see an operating system as just a tool — I see it as a learning environment. Over the past few years, I experimented with Ubuntu, Arch Linux, Debian, RHEL, and Pop!_OS. Each distribution taught me something different about Linux internals, package management, and system architecture.

After all that experimentation, I finally settled on Fedora Linux (KDE Spin) as my daily driver.

This article explains:

  • Why I chose Fedora KDE
  • How I configured it
  • Why I use Podman instead of Docker
  • How this setup supports my development and DevOps learning

🐧 What is Fedora?

Fedora Linux is a community-driven Linux distribution sponsored by Red Hat.

Key characteristics:

  • Upstream-first development model
  • Modern Linux kernel versions
  • Strong default security policies
  • Rapid but stable release cycle
  • Close alignment with enterprise Linux

Fedora serves as the upstream for Red Hat Enterprise Linux. This means many technologies used in enterprise environments are first tested and refined in Fedora. For someone aiming toward system engineering or DevOps roles, that alignment matters.

💎 Why KDE Plasma?

I chose the KDE Spin because of KDE Plasma.

Why KDE works for me:

  • Highly customizable UI
  • Advanced window rules and tiling
  • Multiple virtual desktops
  • Lightweight and responsive
  • Deep system configuration panel

As a developer, I constantly:

  • Run multiple terminal sessions
  • Inspect logs
  • Monitor services
  • Switch workspaces quickly

KDE allows precise control without additional extensions.

🛠 My Development Environment

Programming Stack

  • Python (automation & scripting)
  • GCC (C/C++)
  • Git (version control)
  • VS Code (primary IDE)
  • Podman (containerization)

Daily Linux Tools

  • htop – process monitoring
  • neovim – quick terminal editing
  • tmux – persistent sessions
  • dnf – package management
  • systemctl – service control
  • journalctl – system log inspection

Using these daily strengthens:

  • Process-level understanding
  • Service lifecycle management
  • Log analysis skills
  • Debugging ability

📦 Why I Use Podman Instead of Docker

Fedora integrates deeply with Podman. Instead of installing Docker, I use Podman because:

  • Rootless containers by default
  • No background daemon
  • Native systemd integration
  • OCI-compliant

What Makes Podman Architecturally Different?

Unlike Docker:

  • There is no central daemon process
  • Containers run as regular user processes
  • Security model is stronger in rootless mode

This aligns with Fedora's security-first design.

Using Podman helped me better understand:

  • Linux namespaces
  • cgroups
  • Image layering
  • Container networking
  • Process isolation

Rather than just "running containers," I started understanding how they function internally.

📦 Package Management Philosophy

Fedora uses dnf. Compared to:

  • apt (Ubuntu/Debian)
  • pacman (Arch)

I find dnf provides:

  • Clear dependency resolution
  • Stable yet modern packages
  • Less manual breakage compared to rolling release systems

It strikes a balance between innovation and reliability.

🔐 Security & Enterprise Readiness

Fedora ships with:

1️⃣ SELinux Enabled by Default

Security-Enhanced Linux enforces mandatory access controls. Instead of disabling it, I learned to:

  • Interpret audit logs
  • Understand security contexts
  • Work within policy constraints

2️⃣ FirewallD

Zone-based firewall configuration.

These features mirror enterprise Linux deployments.

🧠 What Fedora Taught Me About Linux Internals

Using Fedora daily improved my understanding of:

  • systemd targets and units
  • Boot sequence behavior
  • Kernel updates
  • User and permission models
  • Network configuration
  • Service dependency chains

Linux became less of a GUI OS and more of a structured, controllable system.

🆚 Why Not Ubuntu?

Ubuntu is excellent for beginners and servers. However, for my workflow:

  • I preferred newer kernel versions
  • I wanted KDE as a primary experience
  • I wanted closer exposure to Red Hat–style systems

Fedora KDE felt cleaner, more configurable, and more aligned with my long-term goals.

🚀 Career Impact

Switching to Fedora KDE + Podman made me:

  • More confident with terminal workflows
  • Comfortable debugging services
  • Familiar with container internals
  • Better prepared for DevOps and system roles

It transformed Linux from a desktop environment into a professional engineering platform.

🎯 Who Should Consider Fedora KDE?

  • Engineering students serious about Linux
  • Developers exploring containers
  • DevOps learners
  • Anyone wanting modern packages without full rolling-release maintenance

📈 Final Thoughts

Fedora KDE Plasma provides:

  • Modern software
  • Enterprise alignment
  • Strong security defaults
  • Deep customization
  • Clean development workflow

For a Computer Engineering graduate building system-level expertise, it offers the right balance between experimentation and stability.

It's not just my operating system. It's my engineering lab.
